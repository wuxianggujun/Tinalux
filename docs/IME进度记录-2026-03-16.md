# IME 进度记录 - 2026-03-16

## 本轮结论

本轮 IME 线的重点不是继续扩验证，而是把当前实际状态记录清楚。

当前结论如下：

- Windows IME 路径可用，相关回归通过
- macOS 文本输入桥接已接入统一组合态事件链
- Linux/X11 代码链路已经接通
- Linux/X11 **尚未做真机运行验证**

这意味着 Linux/X11 目前的状态应表述为：

> **代码已接通，运行时未验证**

这不是“功能未实现”，也不是“已经验证完成”，而是中间状态。

## 已完成内容

### 1. 平台依赖固定

项目已直接使用仓库内本地 `3rdparty/tina_glfw`，
不再依赖顶层 CMake 的在线下载方式来拿桌面窗口后端。

### 2. X11 原生 IME callback 已接通

`tina_glfw` 的 X11 原生接口已经补齐：

- `glfw3native.h` 新增 `glfwSetX11TextInputCallback`
- X11 初始化优先选择 `XIMPreeditCallbacks`
- X11 backend 已实现 `preedit start/draw/caret/done`
- 提交字符时若 preedit 仍然活着，会主动补发 end，避免组合态残留

### 3. Tinalux 统一事件链已接通

Linux/X11 回调已经进入 Tinalux 自己的统一文本组合事件链：

- `GLFWWindow` 接收 X11 text input callback
- 平台层转成 `TextCompositionStart / Update / End`
- `TextInput` 已消费组合态文本、caret 和 target range
- 现有 smoke test 已覆盖统一事件链的基础行为

## 已验证内容

当前已确认通过的内容：

- `TinaluxUIEventSmoke`
- `TinaluxUITextInputSmoke`
- Windows 侧回归干净

## 明确未验证内容

以下内容 **没有** 在 Linux 真机上闭环：

- X11 preedit callbacks 的真实运行行为
- 不同输入法对 target/highlight 的反馈差异
- 候选词窗口位置与 spot 更新的实际表现
- preedit start/update/end 顺序在不同 IM 下的差异

因此，当前不能把 Linux 写成“完整支持且已验证”，
只能写成“代码已接通，未做 Linux 真机验证”。

## 当前决策

当前阶段 **不继续推进 Linux 构建验证或真机验证**。

也就是说，Linux/X11 IME 这条线先停在当前状态：

- 功能实现先保留
- 风险明确记录
- 不把“未验证”伪装成“已完成”

## 下一步建议

如果你问“接下来最该做什么”，我的建议很明确：

### 建议 1：先不要再抠 X11 backend

原因很简单：

- 代码链路已经接上了
- 再往下抠会越来越依赖 Linux 真机反馈
- 你现在已经明确说了，这一轮不需要验证

在这种前提下，继续改 X11 细节，收益很低，猜测成分会越来越高。

### 建议 2：把重心切回上层功能

更值得继续做的是：

1. 富文本能力
2. 文本编辑体验完善
3. 更完整的文本控件能力

原因是这些工作不依赖 Linux 真机，
而且能直接复用已经打通的 `TextCompositionEvent` 链路。

### 建议 3：把 Linux IME 当成“待未来补闭环”的平台项

后面只有在满足下面条件时，再回头处理它才划算：

- 有 Linux/X11 真机或稳定远程环境
- 需要处理某个具体输入法的实际问题
- 有明确 bug 现象，而不是继续预防性猜测

## 一句话建议

**现在别再围着 Linux/X11 IME 打转了，先回到上层文本能力和 UI 功能本身。**
