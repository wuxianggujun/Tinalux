## 核心结论

### 1. 事件系统不是“Phase 3 未完成”

外部分析里提到的几项关键缺口，源码里大多已经实现：

- **三阶段事件分发已实现**
  - 实现在 `UIContext::dispatchEvent()` 和局部辅助函数 `dispatchAlongPath()`
  - 路径顺序为 `capture -> target -> bubble`
  - 键盘事件也会沿焦点路径走三阶段分发，不是直接只打给叶子
- **命中测试已实现**
  - `Container::hitTest()` 会反向遍历子节点并递归命中
  - `Widget::hitTest()` / `hitTestGlobal()` 已区分本地坐标与全局坐标
- **焦点管理已实现**
  - `focusedWidget_`、`setFocus()`、`advanceFocus()` 已在 `UIContext` 中落地
  - `Tab / Shift+Tab` 焦点循环已经接通
  - 鼠标按下时会沿父链查找最近的可聚焦控件
- **CursorEnter 回调已接通**
  - `GLFWWindow` 构造函数里已注册 `glfwSetCursorEnterCallback`
  - `onCursorEnter()` 会派发 `MouseEnter` / `MouseLeave`
  - `UIContext` 会维护 `hoveredWidget_` 并合成控件级别进入/离开事件

这里有一个很重要的结构变化：  
历史文档和外部分析常写成 `Application::dispatchEvent()`，但当前源码里这套能力已经下沉到 `UIContext`，`Application` 主要负责壳层、窗口、渲染循环和状态同步。

### 2. “UI 还没活起来”的判断已经过时

从源码和测试看，交互链路已经不仅限于基础事件：

- `Button` 已支持 hover / pressed / focused 状态和键盘激活
- `TextInput` 已支持焦点、文本输入、选择、拖拽、失焦清理、IME 活跃态同步
- `Overlay` / `Dialog` 已存在，且 `hitTestTopLevel()` 会优先命中 overlay 树
- `event_smoke.cpp` 已覆盖：
  - capture / target / bubble 顺序
  - 键盘事件三阶段分发
  - 鼠标进入/离开
  - Tab 焦点循环
  - 点击聚焦
  - overlay 优先级

结论是：**Phase 3 不是当前主瓶颈，事件链路已经打通。**

### 3. 渲染抽象已收口，多后端已实现（OpenGL + Vulkan）

当前公开渲染 API 已经不再直接暴露核心 Skia 类型，使用的是：

- `Canvas`
- `RenderContext`
- `RenderSurface`
- `Image`

UI 层与上层业务层已经完全不感知 Skia 细节和具体后端选择。

**多后端实现已经落地**，具体现状：

- `rendering::Backend` 枚举定义了 `Auto / OpenGL / Vulkan / Metal` 四种后端
- `rendering::createContext(const ContextConfig&)` 统一入口，按 `config.backend` 分发到具体实现
- `ApplicationConfig` 新增 `backend` 字段，用户可指定后端或使用 `Auto`
- `BackendPlan` 类管理候选后端排序和 fallback 策略：
  - 非 Apple 平台 Auto 优先级：Vulkan → OpenGL
  - Apple 平台 Auto 优先级：Metal → Vulkan → OpenGL
- `Application::tryInitializeBackend()` 逐个尝试候选后端
- `Application::tryPromoteNextBackend()` 运行时 surface 创建失败时自动降级

**OpenGL 后端**（完整可用）：
- `createOpenGLContextImpl()` 通过 `GrDirectContexts::MakeGL()` 创建 Skia Ganesh GL context
- `createOpenGLWindowSurface()` 包装默认 framebuffer 为 `SkSurface`
- `flushFrame()` 调用 `skia->flushAndSubmit()`

**Vulkan 后端**（完整可用）：
- `createVulkanContextImpl()` 创建 VkInstance，加载所有 bootstrap 入口点
- `ensureVulkanDevice()` 枚举物理设备、选择队列族（graphics + present）、创建逻辑设备
- `tryCreateSkiaVulkanContext()` 延迟创建 Skia Ganesh Vulkan direct context
- `createVulkanSwapchain()` 完整的 swapchain 生命周期管理：
  - surface format 选择、present mode 选择、composite alpha 选择
  - swapchain image 获取、per-image SkSurface 包装、semaphore 同步
- `acquireVulkanSwapchainImage()` 帧获取 + semaphore wait
- `flushFrame()` Vulkan 路径：flush with signal semaphore → queue present → 自动 acquire 下一帧
- `VK_ERROR_OUT_OF_DATE_KHR` / `VK_SUBOPTIMAL_KHR` 正确处理，触发 surface 重建
- Portability enumeration / subset 扩展支持（兼容 MoltenVK）

**Metal 后端**（枚举值和 fallback 候选已就位，实现尚未开始）：
- `createContext()` 和 `createWindowSurface()` 对 Metal 返回空 + 打日志
- `RenderContext::Impl::valid()` 对 Metal 返回 false

**Raster 后端**（离屏渲染可用）：
- `createRasterSurface()` 提供不依赖 GPU 的软件渲染 surface

**平台层 Vulkan 支持**：
- `Window` 基类新增 Vulkan 虚函数：`vulkanSupported()` / `requiredVulkanInstanceExtensions()` / `vulkanGetInstanceProcAddress()` / `vulkanPresentationSupported()` / `createVulkanWindowSurface()` / `destroyVulkanWindowSurface()`
- `GLFWWindow` 完整实现了所有 Vulkan 平台接口
- `WindowConfig` 新增 `GraphicsAPI` 枚举（`OpenGL` / `None`），Vulkan 窗口使用 `GLFW_NO_API`

**Pimpl 隔离**：
- `RenderHandles.h` 把 Skia/Vulkan 内部类型（`VulkanContextState` / `VulkanSwapchainState` / `Canvas::Impl` / `RenderContext::Impl` / `RenderSurface::Impl`）完全隔离在 `src/` 内部
- 公共头文件零 Skia 依赖

**运行时 Hook 机制**：
- `RuntimeHooks.h` 定义了 `CreateWindowFn` / `CreateContextFn` / `CreateWindowSurfaceFn` 三个函数指针
- `runtimeHooks()` 返回全局可替换的 hook 结构体，默认指向真实的 `platform::createWindow` / `rendering::createContext` / `rendering::createWindowSurface`
- `ScopedRuntimeHooksOverride` 提供 RAII 作用域替换，测试结束自动恢复
- `Application::tryInitializeBackend()` 和 `renderFrame()` 中的窗口/上下文/surface 创建全部通过 `runtimeHooks()` 间接调用
- `ApplicationTestAccess.h` 提供测试专用入口，可直接调用 `Application::renderFrame()`

**测试覆盖**：
- `render_backend_smoke.cpp` 用 fake Vulkan loader 模拟了完整的 Vulkan 设备引导流程（instance 创建/销毁、物理设备枚举、队列族选择、逻辑设备创建/销毁、presentation support 查询、portability subset 扩展）
- `backend_plan_smoke.cpp` 覆盖了 BackendPlan 的候选排序、fallback 逻辑、activate/hasFallback 边界
- `application_backend_fallback_smoke.cpp` 验证了两个端到端场景：
  - Auto 启动时 Vulkan context 创建失败 → 自动回退到 OpenGL
  - 运行中 Vulkan surface 重建失败 → `tryPromoteNextBackend()` 触发运行时 fallback 到 OpenGL
- 当前全量 smoke test：**36/36 通过**

结论是：**"多后端架构已完整落地，OpenGL 和 Vulkan 均为生产级实现，运行时 fallback 链路已通过端到端测试验证，Metal 是唯一缺口"**。

### 4. 当前能力面已经明显超过旧文档描述

源码里已经存在，但部分旧文档没有及时反映的能力包括：

- `FlexLayout`
- `GridLayout`
- `ResponsiveLayout`
- `RichTextWidget`
- `IconRegistry`
- `ResourceManager`
- `ResourceLoader`
- `createRasterSurface()`
- `Panel` 组件级渲染缓存

因此，很多把项目仍描述为“Button / TextInput 还依赖未来事件系统”“Flex/Grid/RichText 未开始”的文档，已经不再准确。

## 建议的当前主线

多后端架构已落地，建议把主线调整为：

### 优先级 A：Metal 后端实现 + Android 平台扩展

- Metal 后端是唯一未实现的渲染后端，枚举值和 fallback 候选已就位
- 衔接 `AndroidWindow`（NativeActivity/GameActivity + EGL），为 Android 移植铺路
- 触摸事件映射到 MouseEvent

### 优先级 B：文本后端继续收口

- 目前文本度量仍在 `src/rendering/TextBackend.cpp` 中直接依赖 Skia 字体能力
- 可以继续把文本相关能力往更清晰的后端边界上收拢

### 优先级 C：平台验证与构建一致性

- Linux/X11 文本输入链路已有代码，但仍需要真机验证
- 当前 `ctest -N` 能列出 33 个测试，但本次 `Release` 构建里有部分测试目标缺失可执行文件：
  - `TinaluxUIGridLayoutSmoke`
  - `TinaluxUIResourceManagerSmoke`
  - `TinaluxUIResourceLoaderSmoke`
- 这更像是**构建产物同步问题**，不是源码里对应功能不存在

### 优先级 D：代码整理

- `opaqueHandle` / `handleFromOpaque` 模板在 `GLContext.cpp`、`WindowSurface.cpp`、`GLFWWindow.cpp` 三处重复定义，建议提取到共享内部头文件
- Vulkan swapchain 重建在窗口大小不变但 swapchain 失效时（如最小化恢复）可能需要额外 dirty 标记
- `createContext()` 的 `Auto` 分支直接走 OpenGL，与 `BackendPlan` 的 Vulkan 优先策略不一致（虽然 Application 层不会触发此路径）

## 对"当前位置"的修正结论

更准确的说法不是：

> 你处于 Phase 3：事件系统，这是核心中的核心

也不是：

> 当前短板是多后端

而应当是：

> 事件系统和多后端渲染架构均已完成，当前项目处于"Metal 补全 + Android 平台扩展 + 边界收口"阶段。

简化成一句话：

- **事件系统：已打通**
- **多后端渲染：OpenGL + Vulkan 已完整实现，含 fallback 机制**
- **控件与资源能力：已超过基础阶段**
- **当前短板：Metal 后端、Android 平台、平台验证、文本后端边界继续收口**
