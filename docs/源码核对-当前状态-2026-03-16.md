## 结论

### 1. 事件和渲染主链路已经稳定落在 `UIContext + Application`

旧文档里经常把事件核心写成 `Application::dispatchEvent()`，这一点已经过时。

当前真实结构是：

- `Application` 负责窗口、渲染上下文、surface、后端选择、主循环
- `UIContext` 负责 Widget 树、布局、事件派发、焦点、hover、动画和异步资源 pump
- 事件仍然是 capture → target → bubble 三阶段
- overlay 树优先于普通 root 命中

### 2. 多后端渲染现状应表述为“OpenGL / Vulkan 已实现，Metal 未实现”

当前源码里：

- `OpenGL` 路径可创建 Skia Ganesh GL context
- `Vulkan` 路径已覆盖实例、设备、swapchain、surface 重建和同步
- `BackendPlan` 负责候选排序
- `Application::tryPromoteNextBackend()` 负责运行时 fallback
- `Metal` 仍然不是可用后端，最多只能说“接口位置和候选枚举已留好”

### 3. Android 已经从“设计方案”进入“实现中”

Android 当前已不只是文档方案，源码中已经存在：

- `AndroidWindow`
- `AndroidRuntime`
- `AndroidNativeEntry.cpp`
- `AndroidJniBridge.cpp`
- `android/host` 宿主 Kotlin 代码
- `android/tinalux-sdk`
- `android/validation-app`

并且可以确认：

- Android 默认偏好后端是 `OpenGL`
- Vulkan 可以显式指定
- runtime 支持 `attachWindow / detachWindow / renderOnce`
- suspend / resume 会保留 UI 会话并重建窗口与渲染状态

### 4. 主题、样式、资源和文本能力比旧文档描述更完整

以下能力都已经在源码中落地：

- `ColorScheme`、`Typography`、`Spacing`
- `Theme`、`ThemeManager`
- 主题切换动画
- 主题偏好持久化
- `ButtonStyle`、`TextInputStyle`、`SelectionControlStyle`、`SliderStyle`
- `ScrollViewStyle`、`DialogStyle`、`PanelStyle`、`ListViewStyle`、`RichTextStyle`
- `ResourceManager` 的搜索路径和 `@2x/@3x` 解析
- `ResourceLoader` 的异步图片加载与 in-flight 去重
- `RichTextWidget` 的 block role、链接点击、列表、代码块、`maxLines`
- `TextInput` 的 UTF-8、选择、拖拽、IME 组合态、异步图标加载

但也要注意边界：

- 当前是 typed style token 体系，不是通用 CSS / 样式继承引擎
- 没有通用的 Widget Inspector、热重载、布局调试器 UI

### 5. 测试现状不能再写成“全部通过”

这次核对得到两个并存事实：

- 源码中的 [`tests/CMakeLists.txt`](../tests/CMakeLists.txt) 当前定义 `46` 个 smoke target
- 本地 `cmake-build-debug` 下执行 `ctest -N` 只列出 `44` 个测试，并提示大量测试可执行文件缺失

因此当前更准确的表述是：

- 测试覆盖范围已经明显扩大
- 但本地构建目录不是一个干净、完整、可直接采信的验证环境
- 在重新生成和完整构建前，不应写“全量通过”
