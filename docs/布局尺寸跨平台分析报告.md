> 日期：2026-03-19（已二次复核并完成本轮修复）
> 范围：Window → Android 布局转换可行性评估
> 复核结论：原分析总体方向正确，但 `mobile platformScale = 1.0` 不应视为缺陷，Android Native 自动推导 DPI 也不是当前架构目标。

---

## 一、当前尺寸体系概述

Tinalux 的布局系统使用 **逻辑像素（logical pixel）** 坐标：

- `Constraints` 提供 `minWidth/maxWidth/minHeight/maxHeight`（float）
- `Widget::measure()` 返回 `core::Size`，`arrange()` 接受 `core::Rect`
- UIContext 在渲染时通过 `framebufferSize / dpiScale` 计算逻辑尺寸
- 事件坐标同样通过 `1/dpiScale` 归一化到逻辑坐标

**结论：坐标系设计本身是 DPI-aware 的，架构层面可以适配不同密度的屏幕。**

### 建议 4：Mobile Theme Preset — 已实现

```cpp
// Theme.cpp:278-289
Theme Theme::mobile(float systemFontScale) {
    Theme theme;
    theme.platformScale = 1.0f;
    theme.fontScale = sanitizeScale(systemFontScale);
    theme.minimumTouchTargetSize = 48.0f;
    theme.setColors(ColorScheme::dark());
    theme.setTypography(mobileTypography(theme.fontScale));
    theme.setSpacingScale(mobileSpacing());
    theme.setName("mobile");
    return theme;
}
```

包含：
- `mobileTypography()` — 字号缩减（h1: 40→34, h2: 34→30, h3: 28→24 等），再乘以 `fontScale`
- `mobileSpacing()` — md: 16→14, lg: 24→20, xl: 32→24, xxl: 48→32
- 触摸目标保底 48dp
- fill-width 语义

**状态：已完成**

### 建议 6：Android dpiScale 传递 — 按既定架构已完成

**已实现：**
- `AndroidRuntime::attachWindow(void* nativeWindow, float dpiScale)` 接受 dpiScale 参数
- `sanitizeAndroidDpiScale()` 防御性校验，无效值回退到 1.0f 并记录警告

**复核说明：**

- 当前正式方案就是由 Kotlin/Java 宿主层传入 `DisplayMetrics.density`
- Native 层负责 sanitize 和兜底告警，不再伪造自动推导逻辑

**状态：已完成** — 这不是“少做了一步”，而是最终确认后的架构边界

## 三、复核修正与本轮新增收口

### 修正 1：`mobile platformScale = 1.0` 不是缺陷

- `platformScale` 用于表达平台级组件尺寸系数，不等同于 Android 物理 DPI
- 当前 mobile preset 维持 `1.0f` 是有意为之，核心触控保障来自 `minimumTouchTargetSize = 48dp`
- 未来如果要单独做 tablet / large-screen preset，再基于 `platformScale` 拆分即可

### 修正 2：Android Native 自动读取 DPI 不是当前目标

- 现有方案已经明确为“宿主层提供 `dpiScale`，native 做 sanitize”
- 这比在 native core 内部混用 `ANativeWindow`、`AConfiguration` 和宿主显示指标更稳定
- 因此文档里“Native 自动获取 DPI 未实现”应改判为“非目标项”
