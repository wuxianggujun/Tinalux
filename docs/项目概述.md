# Tinalux 项目概述

> 更新时间：2026-03-23

## 项目定位

Tinalux 是一个基于 Skia 的跨平台自绘 UI 框架。按钮、输入框、面板、富文本和对话框都由框架自行布局、命中测试和绘制，不依赖系统原生控件。

当前代码已经不是“只有桌面窗口和少量控件”的早期状态，而是一个包含多后端渲染、主题、资源、文本输入和 Android 运行时骨架的可运行框架。

## 核心模块

- `Core`：事件、几何类型、键码、日志。
- `Platform`：`Window` 抽象、桌面 `GLFWWindow`、Android `AndroidWindow`。
- `Rendering`：Skia 上下文、窗口 surface、离屏 surface、图片与缓存、多后端封装。
- `UI`：Widget 树、布局、控件、主题、动画、资源、富文本、文本输入。
- `Markup`：声明式布局 DSL、Parser、AST、LayoutBuilder、LayoutLoader、ViewModel 绑定与结构重建。
- `App`：`Application`、`UIContext`、后端选择与 fallback、主循环、运行时状态同步。

依赖关系仍然是星形结构：`App` 负责组装，其他模块尽量彼此隔离。

## 当前实现摘要

### 渲染与平台

- `rendering::Backend` 已包含 `Auto / OpenGL / Vulkan / Metal`
- `OpenGL`、`Vulkan` 已实现，`Metal` 在 Apple 平台已有实现
- `BackendPlan` 会在启动和运行时 surface 恢复时执行候选排序与 fallback
- 桌面端窗口实现基于仓库内固定的 `3rdparty/tina_glfw`
- Android 已具备 `AndroidWindow + AndroidRuntime + JNI/C ABI + 宿主层 Kotlin`

### UI 与交互

- 约束布局：`Constraints`、`VBoxLayout`、`HBoxLayout`
- 高级布局：`FlexLayout`、`GridLayout`、`ResponsiveLayout`
- 事件系统：capture → target → bubble 三阶段，含 overlay 优先命中、焦点管理、鼠标捕获、hover 追踪
- 文本：`Label`、`ParagraphLabel`、`RichTextWidget`、`TextInput`
- 控件：`Button`、`Checkbox`、`Radio`、`Toggle`、`Slider`、`ScrollView`、`ListView`、`Dialog`、`Dropdown`、`ProgressBar`、`ImageWidget`
- 主题：`Theme`、`ThemeManager`、`ColorScheme`、`Typography`、`Spacing` 和多种 typed style struct
- 资源：`IconRegistry`、`ResourceManager`、`ResourceLoader`
- 动画：`AnimationSink`、`AnimationScheduler`，并已接入部分控件 hover / focus 动画

### Markup 与声明式布局

- `Parser + Lexer + AST` 已支持无 `@` 关键字语法的 `style`、`import`、`res(...)`、`component`、`Slot`
- `LayoutBuilder + LayoutLoader` 已支持 `Dialog`、`ScrollView`、`Dropdown`、`ImageWidget` 等节点装配
- `ViewModel` 已从扁平键值表扩展为 `Scalar / Object / Array` 树状模型
- `${model.xxx}` 保持运行时属性 binding；`if / elseif / else / for` 和 `${item.xxx}` 这类局部作用域由构建阶段展开
- `${someId.someProperty}` 可用于普通属性和样式绑定，但当前不参与 `if / for` 结构重建
- `ViewModel` 结构路径变化会触发整树重建，重建后会重新挂回属性 binding 和交互回调

## 当前平台状态

- Windows：桌面主链路已实现
- macOS：桌面主链路已实现，含 Cocoa 文本输入桥接
- Linux/X11：代码链路已接通，缺少真机验证，尤其是 IME 和实际运行行为
- Android：运行时和宿主接线已实现，但仍需要真机编译、渲染、输入和生命周期验证

## 构建说明

桌面端默认依赖以下本地源码目录：

- `3rdparty/skia`
- `3rdparty/spdlog`
- `3rdparty/tina_glfw`

典型桌面构建流程：

```bash
cmake -B build -S .
cmake --build build
```

Android 侧当前还提供：

- `scripts/build_android_native.ps1`
- `scripts/stage_android_validation_artifacts.ps1`
- `android/tinalux-sdk`
- `android/validation-app`

## 测试现状

- 当前 Windows / `pwsh` / Vulkan 头可用的工作区里，`ctest -N --test-dir cmake-build-debug -C Debug` 列出 `78` 个测试
- `ctest --test-dir cmake-build-debug -C Debug --output-on-failure --timeout 60 -j 4` 已在本工作区跑通 `78/78`，真实耗时约 `114.04s`
- `TinaluxMarkupLayoutControlFlowSmoke` 已覆盖 `if / elseif / else / for`、局部作用域、数组子路径更新、结构重建和重建后交互重挂
- 测试覆盖面已经扩展到渲染、布局、控件、主题、资源、Android 生命周期
- 当前工作区里的 `cmake-build-debug` 已是可采信的验证环境
- 这次收敛同时修复了普通 `pwsh` 环境下 Ninja/MSVC 对标准头搜索不稳定的问题

## 当前主要缺口

- Android 仍缺真机验证和更完整的输入/手势能力
- Linux/X11 仍缺运行时闭环验证
- CI 侧已开始固化 Linux / Windows / Android 脚本级验证，但仍缺更完整的平台矩阵和真机闭环
