# 声明式布局方案分析

> 日期：2026-03-22
> 问题：Tinalux 是否需要引入 XAML 类声明式布局文件？

> 当前仓库现状补充：项目已经落地一套轻量 markup DSL，支持运行时注册控件、`@style`、样式继承、inline `style: { ... }`、`@import`、`@res("...")`、带参数的 `@component`、`Slot`、`${model.xxx}` 单向/双向属性绑定、树状 `ViewModel`、`@if / @for` 控制流，以及基于结构路径失效的整树重建；它已经是当前主线实现，不再只是一个备选设想。

## 结论

**当前阶段不建议引入 XAML 类外部布局文件。** 推荐继续演进现有 markup DSL，而不是切回新的 Builder DSL 主线。

---

## 一、当前 UI 构建方式

目前 UI 已经进入“双轨”状态：复杂交互页面仍可直接写 C++20，静态结构、样式和绑定明确的界面则可以走现有 markup DSL。

```cpp
auto column = std::make_shared<Container>();
auto layout = std::make_unique<VBoxLayout>();
layout->spacing = 12.0f;
column->setLayout(std::move(layout));

auto button = std::make_shared<Button>("Sign In");
button->setIcon(loginIcon);
button->onClick([&] { /* handler */ });
column->addChild(button);
```

一个中等复杂度的表单页面约 50-200 行 C++ 代码。

### 已有的抽象能力

- **布局**: VBox / HBox / Flex / Grid / Responsive（CSS Flexbox 风格）
- **主题**: Theme::dark() / light()，全局 + 单控件覆盖
- **样式**: StateStyle\<T\> 支持 normal/hovered/pressed/focused/disabled
- **富文本**: RichTextBuilder 链式 API
- **动画**: 属性动画 + 缓动曲线
- **虚拟列表**: ListView + itemFactory

---

## 二、XAML 类方案需要什么

一个完整的 XAML 风格系统至少包含以下子系统：

| 子系统 | 说明 | 工作量 |
|--------|------|--------|
| XML/JSON 解析器 | 解析布局文件为 DOM 树 | 中 |
| 类型注册表 | 字符串 → C++ 类型映射（模拟反射） | 高 |
| 属性绑定 | 字符串属性名 → setter 调用 | 高 |
| 值转换器 | 字符串 → Color / float / enum | 中 |
| 事件绑定 | 布局文件中引用 C++ 回调 | 高 |
| 数据绑定 | 单向/双向属性同步（ViewModel 模式） | 极高 |
| 资源/样式引用 | `{StaticResource}` / `{Binding}` 语法 | 高 |
| 热重载（可选） | 修改文件后实时刷新 UI | 中 |

### 核心难点：C++ 没有运行时反射

XAML 在 WPF/UWP/Avalonia 中依赖 .NET 反射系统：

```xml
<Button Content="Click" Background="Blue" Click="OnClick"/>
```

运行时通过反射找到 `Button` 类、调用 `set_Content()`、解析颜色、绑定事件。C++ 无法做到这一点，必须手写注册宏：

```cpp
REGISTER_WIDGET(Button)
    PROPERTY(text, &Button::setText)
    PROPERTY(icon, &Button::setIcon)
    EVENT(onClick, &Button::onClick)
END_REGISTER
```

每新增/修改一个控件都要同步更新注册表，维护成本高。

## 三、不建议的理由

### 1. 工程量与收益不对等

XAML 子系统的总代码量可能超过现有整个 UI 框架。而收益主要是"写 UI 更快"——但目前控件种类已丰富，C++ API 也足够清晰。

### 2. 与学习目标冲突

项目目标是学习 GUI 内部机制（事件分发、布局算法、文字排版、GPU 管线）。XAML 解析器属于"开发工具链"范畴，不是 GUI 核心知识。

### 3. 运行时成本

外部布局文件意味着运行时解析、查表、间接调用，对性能敏感的自绘 UI 不理想。

### 4. 调试困难

XML 中的拼写错误只能在运行时发现，C++ 代码中的错误在编译期就能捕获。

## 四、替代方案对比

### 方案 A：继续演进现有 markup DSL（推荐，当前主线）

在当前仓库里，这条路线已经不是“从零开始的新方案”，而是沿着现有 `Lexer / Parser / AST / LayoutBuilder / LayoutLoader / ViewModel` 继续往前推：

- 补齐更完整的数据绑定和序列化
- 完善样式系统和资源引用
- 继续扩展控制流、组件复用和开发工具
- 逐步推进热重载、语言服务和可视化工具

**优点：**
- 不推翻当前已落地的 DSL 和测试资产
- `@style / @import / @component / Slot / binding / @if / @for` 已经有真实实现和 smoke
- 运行时装配链路已经接入主题、控件注册、`ViewModel` 和结构重建
- 后续增量工作能直接沉淀到现有模块里

**缺点：**
- 语言设计历史包袱比从零造一个 DSL 更重
- 结构重建当前仍是整树 rebuild，不是增量 patch
- 热重载、LSP、可视化编辑器还没做

### 方案 B：C++ Builder DSL（可选的语法糖，不再是主线）

用 C++ 模板和运算符重载实现声明式语法糖：

```cpp
auto ui = VBox(spacing(12), padding(16), {
    Label("Welcome back"),
    TextInput("email").icon(userIcon).placeholder("name@company.com"),
    TextInput("password").obscured(true),
    HBox(spacing(8), {
        Button("Sign In").primary().onClick([&]{ signIn(); }),
        Button("Clear").onClick([&]{ clear(); }),
    }),
});
```

**优点：**
- 零运行时开销，编译期展开为现有 make_shared + addChild
- 编译期类型检查，拼错属性名直接报错
- 不需要解析器、反射、注册表
- 渐进式引入，可与现有 API 共存
- 工作量小（约 200-500 行模板代码）

**缺点：**
- 不支持热重载
- 仍然是 C++ 代码，非技术人员无法编辑

**参考：** Flutter 的 Widget 组合风格、Dear ImGui 的即时模式、Slint 的 C++ 宏

### 方案 C：JSON/TOML 布局（控件稳定后可考虑）

```json
{
  "type": "VBox",
  "spacing": 12,
  "children": [
    { "type": "Label", "text": "Welcome" },
    { "type": "Button", "text": "Sign In", "style": "primary" }
  ]
}
```

**优点：**
- 格式简单，解析容易（nlohmann/json 等库支持好）
- 可实现热重载
- 非程序员可编辑简单界面

**缺点：**
- 仍需类型注册表
- 事件绑定困难（JSON 无法表达回调）
- 不如 C++ DSL 表达力强

**适合场景：** 静态布局（对话框、设置页），不涉及复杂交互逻辑

### 方案 D：Lua 脚本绑定（游戏引擎集成时）

```lua
local ui = VBox { spacing = 12,
    Label "Welcome",
    Button "Sign In" :onClick(function() signIn() end),
}
```

**优点：**
- Lua 天然适合嵌入 C++ 应用
- 支持热重载
- 游戏行业通用方案
- 比 XAML 系统简单得多

**缺点：**
- 引入 Lua 依赖
- 需要编写绑定层
- 调试比纯 C++ 麻烦

**适合场景：** 未来作为游戏引擎 UI 模块时

### 方案 E：完整 XAML（不推荐）

几乎不建议。除非项目目标转变为"构建一个通用 UI 框架供他人使用"，否则投入产出比极低。

## 五、推荐路线

```text
现在 ──→ 方案 A：继续演进现有 markup DSL
         （保持当前 Parser / Builder / Loader / ViewModel 主线）

按需补充 ──→ 方案 B：C++ Builder DSL
             （作为语法糖，不替代当前 DSL）

游戏引擎集成时 ──→ 方案 D：Lua 脚本绑定
                   （若未来需要更强脚本化与热重载）
```

## 六、Builder DSL 实现草案

核心思路：为每个控件提供链式方法返回 `shared_ptr<Widget>`，为容器提供接受 initializer_list 的工厂函数。

```cpp
// 工厂函数
inline std::shared_ptr<Container> VBox(LayoutProps props,
    std::initializer_list<std::shared_ptr<Widget>> children)
{
    auto container = std::make_shared<Container>();
    auto layout = std::make_unique<VBoxLayout>();
    layout->spacing = props.spacing;
    layout->padding = props.padding;
    container->setLayout(std::move(layout));
    for (auto& child : children) {
        container->addChild(child);
    }
    return container;
}

// 控件链式 API（已部分存在于 RichTextBuilder）
class Button : public Widget {
public:
    Button(std::string text);
    Button& primary();            // 应用 primary 样式
    Button& icon(IconData icon);  // 设置图标
    Button& onClick(std::function<void()> cb);
    // 返回 shared_ptr 的静态工厂，用于 DSL
    static std::shared_ptr<Button> make(std::string text);
};
```

实现工作量预估：200-500 行，可在 1-2 天内完成基础版本。
