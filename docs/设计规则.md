# Tinalux 设计规则

> 本文档记录项目的架构约束、编码规范和设计决策。所有贡献者必须遵循。

---

## 一、模块依赖规则

### 星形依赖 — 最高优先级约束

```
              ┌→ Platform ──→ Core
App (胶水) ───┼→ Rendering ──→ Core + Skia
              └→ UI ─────────→ Core + Skia
```

| 规则 | 说明 |
|------|------|
| **Rendering 不得依赖 Platform** | GL/Vulkan 函数指针通过 `ContextConfig` 参数注入，不 include GLFW 头文件。`createWindowSurface()` 通过 `platform::Window&` 参数接收窗口引用，但 Rendering 模块不 include 具体平台实现头文件 |
| **UI 不得依赖 Rendering** | UI 只面向 `SkCanvas` 和 `core::Event`，不知道 GrDirectContext / VkInstance 的存在 |
| **UI 不得依赖 Platform** | UI 不知道窗口、GLFW、EGL 的存在 |
| **模块间不得互相依赖** | Platform、Rendering、UI 三者互不 include |
| **只有 App 知道所有模块** | App 是唯一的组装层 |

违反这些规则会导致跨平台移植和游戏引擎嵌入变得困难。

## 二、Skia 类型隔离规则

### 目标

将来集成到游戏引擎时，需要用引擎自己的 2D 渲染器替换 Skia。为了让替换代价最小，必须控制 Skia 类型的暴露范围。

### 当前状态

Rendering 模块的公开头文件已完成 Skia 类型隔离：

- `rendering.h` 不再暴露 `sk_sp`、`GrDirectContext`、`SkSurface` 等 Skia 类型
- 公开 API 使用 Pimpl 包装类：`Canvas`、`RenderContext`、`RenderSurface`、`Image`
- Skia/Vulkan 内部类型（`VulkanContextState`、`VulkanSwapchainState` 等）隔离在 `src/rendering/RenderHandles.h`

以下公开头文件中仍暴露了 Skia 基础类型（已知技术债务）：

| 头文件 | 暴露的 Skia 类型 | 影响 |
|--------|-----------------|------|
| `Widget.h` | `SkRect`, `SkSize`, `SkCanvas*` | 所有 Widget 子类都依赖 Skia |
| `Constraints.h` | `SkSize` | 布局系统依赖 Skia |
| `Label.h` | `SkColor` | 控件依赖 Skia |
| `Panel.h` | `SkColor` | 控件依赖 Skia |
| `Theme.h` | `SkColor` | 主题系统依赖 Skia |
| `Layout.h` | `SkRect`, `SkSize` | 布局依赖 Skia |

### 规则

1. **新增头文件不得引入新的 Skia 类型依赖**
   - 新控件的头文件应尽量使用已有的类型（如 `SkColor`、`SkRect`），不引入新的 Skia 类型
   - 如果需要新的 Skia 类型（如 `SkPath`、`SkShader`），只在 `.cpp` 文件中使用

2. **Skia 类型只在 onDraw 的实现中使用**
   - `measure()` 的返回值使用 `SkSize`（已有依赖，暂不改）
   - `arrange()` 的参数使用 `SkRect`（已有依赖，暂不改）
   - `onDraw()` 的参数使用 `SkCanvas*`（已有依赖，暂不改）
   - **新增的绘图逻辑只在 `.cpp` 文件中接触 Skia 的高级 API**（如 `SkParagraph`、`SkPathEffect`）

3. **将来的抽象计划**
   - 当游戏引擎渲染器成型后，引入 `tinalux::ui::Canvas` 抽象层替换 `SkCanvas*`
   - 引入 `tinalux::ui::Color` / `tinalux::ui::Rect` / `tinalux::ui::Size` 替换 Skia 基础类型
   - 提供 `SkiaCanvas` 和 `EngineCanvas` 两个实现
   - **在此之前，不做过早抽象**

4. **`.cpp` 文件可以自由使用 Skia**
   - 所有 Skia 的高级功能（SkPaint、SkFont、SkRRect、SkParagraph 等）只在 `.cpp` 中使用
   - 头文件中的 Skia 依赖仅限于基础类型（SkRect、SkSize、SkColor、SkCanvas*）

## 三、内存管理规则

### 所有权模型

```
Application (unique_ptr)
    ├── Window (unique_ptr)           — 独占
    ├── GrDirectContext (sk_sp)       — Skia 引用计数
    └── rootWidget (shared_ptr)       — 共享所有权
            ├── child (shared_ptr)    — 父持有子
            │     └── parent_ (裸指针)  — 子引用父，不拥有
            └── layout_ (unique_ptr)  — 独占
```

### 规则

| 规则 | 说明 |
|------|------|
| **优先 `unique_ptr`** | 单一所有者时必须用 `unique_ptr`，不用 `shared_ptr` |
| **`shared_ptr` 用于 Widget 树** | 子组件由父容器通过 `shared_ptr` 持有 |
| **`parent_` 用裸指针** | 子组件引用父组件用裸指针，避免循环引用 |
| **禁止裸 `new`/`delete`** | 所有分配通过 `make_unique` / `make_shared` |
| **lambda 中慎用 `shared_ptr`** | 回调捕获兄弟 Widget 的 `shared_ptr` 时注意生命周期，必要时用 `weak_ptr` |
| **禁止自引用 `shared_ptr`** | Widget 的回调不得捕获自身的 `shared_ptr`，否则永远不会释放 |

### 自引用陷阱示例

```cpp
// ❌ 错误 — 循环引用，永远不会释放
auto button = std::make_shared<Button>("Bad");
button->onClick([button] { button->setLabel("Leaked!"); });

// ✅ 正确 — 用 weak_ptr
auto button = std::make_shared<Button>("OK");
std::weak_ptr<Button> weak = button;
button->onClick([weak] {
    if (auto btn = weak.lock()) {
        btn->setLabel("Clicked!");
    }
});
```

## 四、事件系统规则

### 坐标约定

| 方法 | 坐标系 | 说明 |
|------|--------|------|
| `hitTest(float x, float y)` | **Widget 本地坐标** | (0,0) 是 Widget 左上角 |
| `Event::x / Event::y` | **全局 framebuffer 坐标** | 由 GLFW 回调提供，已 DPI 缩放 |
| `containsGlobalPoint(x, y)` | **全局 framebuffer 坐标** | 内部通过 `globalBounds()` 转换 |
| `bounds_` | **父容器本地坐标** | Widget 在父容器中的位置和大小 |
| `globalBounds()` | **全局 framebuffer 坐标** | 沿 parent 链累积 offset |

### 三阶段派发

所有输入事件（鼠标和键盘）都应走三阶段派发路径：

1. **捕获**（根 → 目标前一个）：依次调用 `onEventCapture()`
2. **目标**：调用 `onEvent()`
3. **冒泡**（目标后一个 → 根）：依次调用 `onEvent()`

任一阶段 `event.handled = true` 或 `event.stopPropagation = true` 时停止。

### GLFW 键值常量

UI 层使用的键值常量应在文件顶部用 `constexpr` 定义并注释 GLFW 原始宏名，将来应迁移到 `core/KeyCodes.h` 统一管理：

```cpp
constexpr int kKeyEnter = 257;       // GLFW_KEY_ENTER
constexpr int kKeyBackspace = 259;   // GLFW_KEY_BACKSPACE
```

## 五、布局系统规则

### 两趟布局

1. **measure（自底向上）**：Widget 在给定 Constraints 内返回期望尺寸
2. **arrange（自顶向下）**：父组件分配最终 bounds

### 规则

| 规则 | 说明 |
|------|------|
| **measure 不得有副作用** | 不修改 `bounds_`，不触发绘制 |
| **arrange 只设置 `bounds_`** | 不触发绘制，但可 `markDirty()` |
| **onDraw 不做布局计算** | 绘制时 `bounds_` 已确定，直接使用 |
| **measure 结果应可缓存** | 相同 Constraints 输入应返回相同结果（将来优化用） |

## 六、渲染规则

### 绘制职责分离

```cpp
// Widget::draw() — 框架调用，处理坐标变换
void Widget::draw(SkCanvas* canvas) {
    canvas->save();
    canvas->translate(bounds_.x(), bounds_.y());
    canvas->clipRect(SkRect::MakeWH(bounds_.width(), bounds_.height()));
    onDraw(canvas);           // ← 子类只实现这个
    canvas->restore();
    dirty_ = false;
}

// Widget::onDraw() — 子类实现，在本地坐标系中绘制
// (0,0) 是 Widget 左上角，无需关心在屏幕上的实际位置
```

### 规则

| 规则 | 说明 |
|------|------|
| **onDraw 中使用本地坐标** | 不要使用 `globalBounds()`，`draw()` 已处理了 translate |
| **修改属性必须 markDirty()** | 所有 setter（setText、setColor 等）在值变化时调用 `markDirty()` |
| **markDirty() 去重** | 已经 dirty 时不重复传播 |
| **setter 做去重检查** | 新值等于旧值时直接 return，不调 `markDirty()` |

## 七、编码规范

### 命名

| 类别 | 规则 | 示例 |
|------|------|------|
| 文件 | PascalCase | `Widget.h`, `GLFWWindow.cpp` |
| 类/结构体 | PascalCase | `Container`, `VBoxLayout` |
| 函数 | camelCase | `addChild()`, `hitTest()` |
| 成员变量 | camelCase 带尾随下划线 | `bounds_`, `visible_`, `parent_` |
| 命名空间 | lowercase | `tinalux::platform` |
| 常量 | k 前缀 + PascalCase | `kKeyEscape`, `kOpenGlMajor` |

### 头文件

- 所有头文件使用 `#pragma once`
- 公开头文件在 `include/tinalux/` 目录下
- 内部头文件在 `src/` 对应模块目录下（如 `src/ui/TextPrimitives.h`）
- 公开头文件不 include 内部头文件

### 前置声明

- 头文件中优先使用前置声明，减少编译依赖
- 只在 `.cpp` 中 include 完整定义

```cpp
// ✅ 好 — Widget.h 中前置声明
namespace tinalux::core { class Event; }
class SkCanvas;

// ❌ 不好 — 在头文件中 include 完整定义
#include "tinalux/core/events/Event.h"
#include "include/core/SkCanvas.h"
```

## 八、游戏引擎集成预留

### UIContext 拆分（已完成）

`Application` 已拆分为 Application（壳）+ UIContext（核心）：

```
Application（独立运行的壳）
    ├── BackendPlan（多后端候选排序 + fallback）
    ├── Window + RenderContext + RenderSurface（渲染资源）
    └── UIContext（核心 UI 逻辑）
            ├── 事件派发（三阶段）
            ├── 布局计算
            ├── Widget 树管理（含 Overlay）
            ├── 焦点管理
            ├── 动画调度
            └── render(Canvas, w, h)
```

预留满足情况：

1. **UI 模块不依赖 Application** — 已满足 ✅
2. **Widget 的 measure/arrange/onDraw 不接触窗口/GL/Vulkan** — 已满足 ✅
3. **事件类型独立于 GLFW** — 已满足 ✅（键值常量已迁移到 `core/KeyCodes.h`）

### Skia 可替换性

将来替换 Skia 时的影响范围：

```
需要改的：
├── 所有 Widget 的 onDraw() 实现    — 用新的 Canvas API
├── TextBackend.cpp                — 用新的文本度量 API
├── Theme.h 中的 SkColor           — 用自定义 Color 类型
└── Widget.h / Constraints.h / Layout.h 中的 SkRect/SkSize — 用自定义类型

不需要改的：
├── 事件系统 (Event, EventDispatcher)
├── 布局算法逻辑 (measure/arrange 的流程)
├── Widget 树管理 (addChild/removeChild/parent)
├── 焦点管理
├── 脏标记传播
├── 多后端选择逻辑 (BackendPlan, Application fallback)
└── Vulkan 平台接口 (Window 基类的 Vulkan 虚函数)
```
