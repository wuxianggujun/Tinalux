# Tinalux 源码导览

> 更新时间：2026-04-05
> 目标：给想读源码的人一张“先看哪里、每层负责什么、主链路怎么走”的地图。

## 1. 先记住这张图

```text
桌面入口
main.cpp
└── app::Application
    ├── BackendPlan
    ├── platform::Window
    ├── rendering::RenderContext / RenderSurface
    └── app::UIContext
        ├── Widget 树
        ├── 事件分发
        ├── 主题运行时
        ├── 动画 / 资源 tick
        └── 渲染

声明式页面入口
.tui 文件
└── markup::Lexer / Parser
    └── AST
        └── LayoutBuilder / LayoutLoader
            └── Widget 树 + LayoutHandle + ViewModel 绑定

Android 入口
android/host (Kotlin)
└── JNI / NativeBridge
    └── app::AndroidRuntime
        └── app::Application
```

如果你脑子里先有这张图，后面再看目录就不会乱。

## 2. 顶层目录怎么看

### `include/`

对外公开头文件。

如果你想先看 API 轮廓，而不是先钻实现，优先从这里读。

### `src/`

项目真正的实现主体，主要分成六层：

- `src/core`
- `src/platform`
- `src/rendering`
- `src/ui`
- `src/markup`
- `src/app`

### `samples/markup/`

不是新 API，而是四种推荐起手模板。

适合你想快速照抄一条可用路径时看。

### `tests/`

主要是 smoke tests。

它们一方面做回归验证，另一方面也是非常好的“最小行为样例”。

### `android/`

Android 相关内容分成三块：

- `host`：Kotlin 宿主层
- `tinalux-sdk`：SDK / AAR 形态封装
- `validation-app`：最小验证工程

## 3. 桌面主链路从哪里看

### 第一站：`main.cpp`

桌面最短入口就是这里。

当前流程非常直接：

1. 读取 `TINALUX_DESKTOP_BACKEND`
2. 创建 `app::Application`
3. 调 `init(...)`
4. 安装 `Theme::dark()`
5. 调 `createDemoScene(...)`
6. `run()`

所以如果你只是想知道“程序怎么启动”，先看 `main.cpp` 就够了。

### 第二站：`include/tinalux/app/Application.h`

这里能看到应用层真正暴露出来的主接口：

- `init`
- `run`
- `pumpOnce`
- `suspendRendering`
- `resumeRendering`
- `setRootWidget`
- `setTheme`

这层是“应用组装器”，负责把平台、渲染和 UI 串起来。

### 第三站：`src/app/Application.cpp`

这里是桌面主链路真正的实现核心。

你应该重点看三件事：

1. 启动时怎么选择 backend
2. 主循环里什么时候处理事件、什么时候渲染
3. 渲染状态丢失后怎么恢复和 fallback

从当前源码可以确认：

- `Application::init(...)` 会读取 `BackendPlan` 的候选列表
- 它会逐个尝试候选后端
- 某个后端失败时，不会立刻退出，而是继续尝试下一个候选

所以 `Application` 不是“只开一个窗口”那么简单，它还承担了运行时恢复和后端回退的职责。

## 4. `UIContext` 负责什么

如果 `Application` 是总装层，那 `UIContext` 就是 UI 运行时中枢。

从 `src/app/UIContext.h` 可以直接看出来，它主要负责：

- 处理事件
- 构建 Widget 树
- 管 root / overlay
- 处理焦点、hover、鼠标捕获
- tick 动画和异步资源
- 请求重绘
- 真正执行渲染

所以读源码时可以这样理解：

- `Application` 负责把窗口、渲染上下文、UI runtime 接起来
- `UIContext` 负责“这棵 UI 树怎么活起来”

如果你要查事件分发、重绘、焦点切换、局部重绘，优先读 `UIContext`。

## 5. UI 层从哪里下手

### 基础骨架

优先看这些文件：

- `include/tinalux/ui/Widget.h`
- `src/ui/Widget.cpp`
- `include/tinalux/ui/Container.h`
- `src/ui/Container.cpp`
- `include/tinalux/ui/Constraints.h`
- `src/ui/Constraints.cpp`

这一层决定了 Widget 树、测量、布局和基础交互的骨架。

### 布局

如果你关心布局系统，直接看：

- `src/ui/layouts/VBoxLayout.cpp`
- `src/ui/layouts/HBoxLayout.cpp`
- `src/ui/layouts/FlexLayout.cpp`
- `src/ui/layouts/GridLayout.cpp`
- `src/ui/layouts/ResponsiveLayout.cpp`

### 控件

常用控件都在：

- `src/ui/widgets/`

比如：

- `Button.cpp`
- `TextInput.cpp`
- `ListView.cpp`
- `Dialog.cpp`
- `RichText.cpp`

### 主题

主题主线比较清晰：

- `src/ui/Theme.cpp`：内建主题定义
- `src/ui/ThemeManager.cpp`：主题切换和运行时绑定

当前源码里已经注册了：

- `Theme::dark()`
- `Theme::light()`

默认主题是 `dark`。

## 6. Markup 主链路从哪里看

如果你只关心声明式页面，不要先从生成代码噪音里硬读。

建议顺序是：

### 第一站：看接口边界

- `include/tinalux/markup/LayoutLoader.h`
- `include/tinalux/markup/ViewModel.h`

这两份头文件能先帮你建立心智模型：

- `.tui` 怎么被加载成 Widget 树
- `LayoutHandle` 持有了什么运行时状态
- `ViewModel` 怎么挂进去

### 第二站：看解析和构建

- `src/markup/Lexer.cpp`
- `src/markup/Parser.cpp`
- `src/markup/LayoutBuilder.cpp`
- `src/markup/LayoutLoader.cpp`

你可以把这一层理解成：

1. DSL 文本先被解析成 AST
2. AST 再被装配成 Widget 树
3. `LayoutHandle` 保存 `idMap`、binding、interaction binding
4. `ViewModel` 变化会驱动属性更新或结构重建

### 第三站：看自动生成入口

构建层入口不在 `src/markup`，而在 `cmake/`：

- `cmake/TinaluxMarkupAutogenTools.cmake`
- `cmake/TinaluxMarkupTools.cmake`

日常开发最重要的就是两个入口：

- `tinalux_add_markup_executable(...)`
- `tinalux_target_enable_markup_autogen(...)`

仓库里的四个样板 target 也就是从这套工具链接出来的。

## 7. Android 这条线怎么读

Android 不是单个目录，而是一条跨目录链路。

### Kotlin 宿主层

先看：

- `android/host/src/main/kotlin/com/tinalux/runtime/TinaluxActivity.kt`
- `android/host/src/main/kotlin/com/tinalux/runtime/TinaluxSurfaceView.kt`
- `android/host/src/main/kotlin/com/tinalux/runtime/TinaluxRendererHost.kt`
- `android/host/src/main/kotlin/com/tinalux/runtime/TinaluxInputConnection.kt`

这层负责：

- Surface 生命周期
- 输入法桥接
- 宿主 Activity 接线
- 将 Android 事件送进 native runtime

### Native runtime

再看：

- `src/app/android/AndroidRuntime.cpp`
- `src/app/android/AndroidJniBridge.cpp`
- `src/platform/android/AndroidWindow.cpp`

这层负责：

- 把 `ANativeWindow` 接成 `Application`
- attach / detach window
- suspend / resume rendering
- 同步剪贴板和输入状态

### SDK 与验证工程

最后看：

- `android/tinalux-sdk`
- `android/validation-app`

这两个目录不是新的 runtime 内核，而是：

- 一个负责打包复用
- 一个负责最小验证

## 8. 如果你想改某类功能，先看哪里

可以按这个速查表找入口：

- 想改桌面启动行为：`main.cpp`、`src/app/Application.cpp`
- 想改后端选择 / fallback：`src/app/BackendPlan.*`、`src/app/Application.cpp`
- 想改桌面窗口接线：`src/platform/glfw/GLFWWindow.cpp`
- 想改渲染后端：`src/rendering/`
- 想改主题：`src/ui/Theme.cpp`、`src/ui/ThemeManager.cpp`
- 想改控件行为：`src/ui/widgets/`
- 想改布局算法：`src/ui/layouts/`
- 想改 Markup DSL：`src/markup/Lexer.cpp`、`Parser.cpp`
- 想改 Markup 装配和绑定：`src/markup/LayoutBuilder.cpp`、`LayoutLoader.cpp`、`ViewModel.cpp`
- 想改 Android 宿主：`android/host/`
- 想改 Android native runtime：`src/app/android/`、`src/platform/android/`

## 9. 推荐源码阅读顺序

如果你是第一次读这个仓库，我建议用下面这个顺序：

1. `main.cpp`
2. `include/tinalux/app/Application.h`
3. `src/app/Application.cpp`
4. `src/app/UIContext.h` / `src/app/UIContext.cpp`
5. `src/ui/Widget.cpp` 和 `src/ui/layouts/`
6. `src/ui/widgets/`
7. 如果你关心声明式页面，再转 `src/markup/`
8. 如果你关心 Android，再转 `android/host/` 和 `src/app/android/`

这样读的好处是：

- 先把主链路搞清楚
- 再看 UI runtime
- 最后再进 DSL 和平台专项

不会一上来就被局部细节淹没。
