# Tinalux 用户快速上手

> 更新时间：2026-04-05
> 目标：给第一次接触 Tinalux 的用户一条最短、最稳的上手路径。

## 1. 这是什么

Tinalux 是一个基于 Skia 的跨平台自绘 UI 框架。

它不是系统原生控件的薄封装，而是自己维护：

- Widget 树
- 布局
- 命中测试
- 绘制
- 主题
- 文本输入

从当前源码状态看，仓库已经具备：

- 桌面窗口主链路
- `OpenGL / Vulkan / Metal` 渲染后端
- 常见控件、主题、资源和文本输入
- 声明式 Markup 页面能力
- Android 原生运行时、宿主层和验证工程

## 2. 你应该先走哪条路径

按你的目标选一条最短路径就够了。

### 我只是想先看到项目跑起来

先走桌面 Demo：

1. 构建 `Tinalux`
2. 跑默认演示界面
3. 再决定要看 UI、渲染，还是 Markup

### 我主要想用 Markup 开页面

先走 Markup 主路线：

1. 跑 `TinaluxRunMarkupMentalModelExamples`
2. 看 [`Markup一页式速查`](./Markup一页式速查.md)
3. 抄 [`samples/markup`](../samples/markup/README.md) 模板

### 我想看 Android 这条线

先理解职责边界：

1. `src/app/android` 和 `src/platform/android` 是 native runtime
2. `android/host` 是 Kotlin 宿主层
3. `android/tinalux-sdk` 是 SDK/AAR 形态封装
4. `android/validation-app` 是最小验证工程

## 3. 构建前准备

桌面端默认依赖以下本地源码目录：

- `3rdparty/skia`
- `3rdparty/spdlog`
- `3rdparty/tina_glfw`

基础要求：

- CMake `3.31+`
- 支持 C++20 的编译器
- 可用的 Skia 构建产物，或使用项目里的 Skia CMake 流程自动构建

如果你在 Windows + MSVC 环境下构建，顶层 `CMakeLists.txt` 已补齐了对部分
终端环境的头文件 / SDK 路径兼容处理。

## 4. 最短桌面运行路径

桌面程序入口在 `main.cpp`。

默认行为是：

1. 创建 `Application`
2. 根据环境变量解析桌面渲染后端
3. 用 `Theme::dark()` 安装默认主题
4. 用 `createDemoScene(...)` 挂载演示界面
5. 进入主循环

直接构建：

```bash
cmake -S . -B build
cmake --build build --config Debug --target Tinalux
```

如果你使用的是单配置生成器，比如 Ninja，可以去掉 `--config Debug`。

如果要手动指定桌面后端，可以设置环境变量：

```bash
TINALUX_DESKTOP_BACKEND=auto
TINALUX_DESKTOP_BACKEND=opengl
TINALUX_DESKTOP_BACKEND=vulkan
TINALUX_DESKTOP_BACKEND=metal
```

源码里的默认逻辑是：

- 未设置时走 `Backend::Auto`
- `Application` 启动时通过 `BackendPlan` 依次尝试候选后端
- 某个后端失败时会继续 fallback

## 5. 最短 Markup 上手路径

如果你真正关心的是“怎么写页面”，不要先钻底层 helper。

先执行这条命令：

```bash
cmake -S . -B build -DTINALUX_BUILD_TESTS=ON
cmake --build build --config Debug --target TinaluxRunMarkupMentalModelExamples
```

这条 target 在测试工程里定义，用来跑两组 Markup mental model smoke。

然后按这个顺序看：

1. [`Markup一页式速查`](./Markup一页式速查.md)
2. [`samples/markup/README.md`](../samples/markup/README.md)
3. [`Markup页面推荐写法`](./Markup页面推荐写法.md)
4. [`Markup常见问题`](./Markup常见问题.md)
5. 最后再按需查 [`Markup高级接口`](./Markup高级接口.md)

普通页面开发，当前推荐主路线是：

- CMake 里用 `tinalux_add_markup_executable(...)`
- 页面构造时直接写 `Page(theme, [&](auto& ui) { ... })`
- 绑定事件时直接用 `ui.xxx.onXxx(...)`
- 页面持有后再通过 `page.ui.xxx` 访问控件代理

## 6. 如果你想编译仓库里的 Markup 模板

先开启样板构建：

```bash
cmake -S . -B build -DTINALUX_BUILD_SAMPLES=ON
```

然后按需编译这些 target：

```bash
cmake --build build --config Debug --target TinaluxMarkupSingleFileSample
cmake --build build --config Debug --target TinaluxMarkupDirectoryScanSample
cmake --build build --config Debug --target TinaluxMarkupScaffoldSingleFileSample
cmake --build build --config Debug --target TinaluxMarkupScaffoldDirectoryScanSample
```

四个 target 分别对应：

- 单文件
- 目录扫描
- 单文件 scaffold
- 目录扫描 scaffold

## 7. 当前平台状态

按源码现状看：

- Windows：桌面主链路已实现
- macOS：桌面主链路已实现，含 Cocoa 文本输入桥接
- Linux/X11：代码链路已接通，但还缺少更多真机验证
- Android：runtime、宿主层和验证工程都已在仓库中

需要注意：

- Android 默认仍以 `OpenGL` 为主
- Vulkan 在 Android 侧是显式 opt-in 路线
- Linux/X11 和 Android 的真实设备闭环仍然比 Windows / macOS 更弱

第一次接触 Tinalux，最稳的顺序是：

- 先把桌面 Demo 跑起来

这样心智负担最低，也最不容易被历史文档和低层接口带偏。
