## 版本控制规则

- 提交命名：采用 `Conventional Commits`，格式为 `<type>(scope): <summary>`
- 推荐类型：`feat`、`fix`、`refactor`、`test`、`docs`、`build`、`ci`
- 版本标记：建议从下一个里程碑开始启用 `SemVer`
- 推荐里程碑标签：
  - 里程碑 A 完成后打 `v0.1.0`
  - Android 真机闭环完成后打 `v0.2.0`
  - Linux/X11 运行时闭环完成后打 `v0.3.0`

## 优先级建议

以下建议采用 `WSJF` 思路排序，优先兼顾交付价值、风险收敛和实现成本。

### 1. Android 真机闭环与输入链路补齐

- 优先级：`P0`
- WSJF 判断：最高
- 目标：跑通 `android/validation-app` 的真实设备验证，补齐 `OpenGL` 默认后端、显式 `Vulkan`、IME、剪贴板、`pause/resume`、`surface detach/attach`
- 预估工期：`6-8` 人日
- 资源需求：
  - `1` 名 C++/跨平台工程师
  - `1` 名 Android 工程师兼职支持
  - `2` 台 `arm64-v8a` 真机，至少覆盖一台 Vulkan 能力较稳定的设备
- 预期产出：设备验证记录、问题清单、最小可复现脚本、更新后的平台结论文档

### 2. 桌面 Smoke 与平台验证矩阵固化到 CI

- 优先级：`P0`
- WSJF 判断：高
- 目标：把当前本地可跑通的桌面 smoke 路径、Linux/X11 构建验证、Android 构建校验脚本纳入统一 CI 入口
- 预估工期：`3-5` 人日
- 资源需求：
  - `1` 名 C++/DevOps 工程师
  - GitHub Actions runner 与缓存策略
  - Windows / Linux runner；Android 可先从 validate-only 路径开始
- 预期产出：统一工作流、失败回溯入口、可复用的验证命令与产物归档

### 3. Linux/X11 运行时与 IME 闭环验证

- 优先级：`P1`
- WSJF 判断：高
- 目标：在真实 Linux/X11 环境下验证窗口创建、渲染、IME preedit、不同输入法差异，并将结论收敛到文档
- 预估工期：`4-6` 人日
- 资源需求：
  - `1` 名 C++ 平台工程师
  - `1` 套 Linux/X11 测试环境
  - 至少 `2` 种输入法进行交叉验证
- 预期产出：运行时验证记录、IME 差异矩阵、平台状态由“代码已接通”升级为“已验证”

### 4. 移动端触摸与手势系统

- 优先级：`P1`
- WSJF 判断：中高
- 目标：补齐触摸事件语义、滑动与缩放等基础手势，让 Android 端从“可运行”升级到“可交互”
- 预估工期：`7-10` 人日
- 资源需求：
  - `1` 名 UI/交互工程师
  - `1` 名平台工程师支持事件桥接
  - Android 真机回归验证
- 预期产出：触摸事件模型、手势识别基础设施、对应 smoke 或设备验证用例

## 建议的下一步

1. 先完成 Android 真机闭环，尽快把“代码已接通”升级为“可验证发布”
2. 同步把本地 smoke 路径固化到 CI，避免后续文档与真实基线再次漂移
3. Android 与 CI 收敛后，再推进 Linux/X11 运行时验证和移动端触摸系统
